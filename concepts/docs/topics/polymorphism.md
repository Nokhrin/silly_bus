# Полиморфизм  
  
## Аксиоматическое определение | Идея | Что это?  
Свойство системы типов данных, обеспечивающее взаимозаменяемость значений надтипа и подтипа с помощью применения единых для надтипа и подтипа сигнатур методов

  
## Прагматическое определение | Воплощение идеи | Для чего это?  
Обобщение типа данных, позволяющее подтипу переопределять тело метода надтипа с сохранением сигнатуры переопределяемого метода
для чего: изменение/добавление логики/поведения с помощью добавления подтипа данных

### Повышение надежности выполнения
(полиморфизм как способ абстрагирования)

[Пример - Хрупкость API RandomAccessFile](../../src/main/java/org/example/abstraction/FileOperations.java)

### Поддержка выполнения требования | Соблюдения контракта | Работоспособности интерфейса
полиморфизм создает связь классов с помощью системы типов


### Добавление методов
(полиморфизм как способ реализации open/closed principle)
ситуация: Требования к системе изменились - контракт дополнили
профит полиморфизма: уменьшение количества строк кода: +1 тип -> +1 класс, вместо +N точек вызова, которые нужно исправить

#### пример
Задача - выбор сериализатора
для добавления сериализации нового формата создаю **новый класс**, не трогаю существующие *.class  
[Динамический выбор / Полиморфизм](../../src/main/java/org/example/polymorphism/serializables/SerializableDynamic.java)

#### контрпример
для добавления сериализации нового формата меняю **существующий класс**, необходимо перекомпилировать систему  
[Статический выбор / НеПолиморфизм](../../src/main/java/org/example/polymorphism/serializables/SerializableStatic.java)


### Однократная верификация значений  
(полиморфизм как способ реализации inheritance)

профит полиморфизма: логика проверок - в корневом классе, добавление методов в интерфейс не потребует повтора логики проверок

ситуация: множество критичных бизнес-требований - баланс должен быть положительным после операции, сумма должна быть неотрицательной

## Вопросы

### 1
> каким образом старый *.class узнает о новом без изменения и перекомпиляции?

новый метод сохраняет сигнатуру, обработка которой описана в существующем .class
новый метод сохраняет сигнатуру, но содержит отличную реализацию тела метода
сигнатура = имя метода + последовательность параметров + тип возвращаемого значения + типы параметров

в добавляемом классе программист определяет сигнатуру метода, которая определена в существующем классе 
с точки зрения существующего класса изменений не происходит - сигнатура соответствует ожидаемой
