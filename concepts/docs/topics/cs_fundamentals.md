### 9
Допустим, у нас есть int a = 5; int b = 10; int c = a + b;
Вариант А: C (Native x86_64 Assembly)
Компилятор решает, где хранить a, b, c. Допустим, они в памяти (на стеке), а для сложения используются регистры.

```assembly
; Представим, что RBP указывает на текущий фрейм стека
MOV RAX, [RBP - 4]   ; 1. Загрузить 'a' из памяти (смещение -4) в регистр RAX
MOV RBX, [RBP - 8]   ; 2. Загрузить 'b' из памяти (смещение -8) в регистр RBX
ADD RAX, RBX         ; 3. Сложить RAX и RBX (результат в RAX)
MOV [RBP - 12], RAX  ; 4. Записать результат из RAX в память (переменная 'c')
```

## лямбда как указатель на функцию и объект

понимание высказывания `void (*func_ptr)(int) = &my_function;`

конструкция инициализирует переменную `func_ptr`
`func_ptr` есть указатель (`*` перед `func_ptr`) на функцию `my_function`
`func_ptr` является переменной, содержащей адрес первой инструкции функции `my_function`
`my_function` принимает один аргумент типа `int`
`my_function` не возвращает значения - сигнатура возвращаемого типа `void`

пример вызова:
`(*func_ptr)(10)`
будет вызвана функция `my_function` над аргументом `10`

инициализация лямбды в java создает анонимный класс и его экземпляр, 
полем которого является копия значения переменной, переданной лямбде
=> значение переменной записывается в поле объекта лямбды - это захват переменной | замыкание
=> лямбда в java имеет состояние
```java
public class Main {
    public static void main(String[] args) {
        String name = "lambda";

        Consumer<String> lambda = (n) -> System.out.println(n + ", " + name); 
        
        lambda.accept(("hello"));
        
        // name = "world";
        // изменение значения захваченной переменной вызовет ошибку
        // java: local variables referenced from a lambda expression must be final or effectively final
    }
}
```
=> лямбда может иметь поля, имеет метод, может быть передана как аргумент
