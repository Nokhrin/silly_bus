# Валидатор связности документации  

> Проверка связности docs: проверка ссылок и цитат. Рекурсивный скан, отчет. Гарантия целостности

Цель автоматизации:  
Снижение трудозатрат на верификацию зависимостей документации с O(N×M)O(N×M) (ручная проверка NN файлов с MM ссылками) до O(N+M)O(N+M) (автоматический обход графа). Исключение человеческого фактора при проверке актуальности цитат и доступности ресурсов  
Функциональные требования:  
  
Входные данные: Корневая директория, список расширений (.md, .puml, .tex, .ebnf)  
Извлечение: Парсинг содержимого файлов (UTF-8) для выявления URI (http, ftp, file) и блоков цитирования с метаданными источника  
Верификация:  
Для URI: проверка статуса ответа (HTTP HEAD) или существования файла в ФС  
Для цитат: вычисление контрольной суммы (CRC32/SHA-256) указанного диапазона строк исходного файла и сравнение с задекларированным значением  
Отчетность: Генерация структурированного отчета (JSON/Console) со статусом каждой связи и кодом ошибки при неудаче  
Нефункциональные требования: Идемпотентность (отсутствие модификации входных файлов), логирование ошибок, обработка исключений сети/ФС  

## решаемая проблема  
Дрейф документации относительно исходного кода или внешних ресурсов («битые ссылки», устаревшие примеры)  

## возможное решение  
Автоматизированная проверка соответствия декларированных зависимостей между единицами данных их фактическому состоянию в распределенной среде  
Алгоритм: Построение ориентированного графа зависимостей, обход графа, проверка существования узлов и соответствия содержимого посредством хеш-функции  
Требования:  
- состояние: Граф зависимостей не содержит недоступных узлов (404, отсутствующие файлы)  
- поведение: Система должна завершать обход даже при ошибках сети, фиксируя их в отчете  
Ограничения:  
- невозможность валидации бинарных файлов документации (.pdf, .doc, .png и т.п.)

## вопросы
Как формально отделить «временную недоступность сети» от «битой ссылки» в метриках надежности?  
Если система сообщает об ошибке валидации, но документация визуально корректна для читателя, является ли ошибка валидации дефектом системы или дефектом документации?  

> вторичная проблема - валидация синтаксиса markdown - некоторые строки требуют жесткого переноса строки (два пробела в конце строки) для корректной интерпретации абзаца  
> пример:  
> некоторые строки  
> требуют жесткого переноса строки  
> (два пробела в конце строки)  
> для корректной интерпретации абзаца
> 
> некоторые строки
> требуют жесткого переноса строки
> (два пробела в конце строки)
> для корректной интерпретации абзаца
> 
> возможно добавить регуляркой ко всем строкам два пробела в конце, но такое решение приведет к проблеме  
> жесткий перенос в конце строки заголовка ломает заголовок как ссылку, ломает чтение заголовка, следующего за строкой, содержащей только два пробела  
> т.е. жесткий перенос нельзя применить ко все строкам  
> смысл валидации - убедиться, что жесткий перенос применен целесообразно  
> существующее решение данной задачи - валидация глазами  
> существующее автоматизированное решение данной задачи не удалось найти, например, плагин markdown для IDEA не имеет такой функциональности
