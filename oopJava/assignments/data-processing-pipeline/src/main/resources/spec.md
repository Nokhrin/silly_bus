# Конвейер обработки данных  

Цель: показать, что отказ от лямбд ведет к экспоненциальному дублированию классов-адаптеров.  
Требование: Использовать все синтаксические формы JLS (выражение, блок, метод-референс, конструктор-референс).  


## Бизнес-проблема  
сверка и аудит финансовых транзакций  

Ввод данных: Ежедневный объем поступлений составляет 10 000+ записей из ERP  
Ручная обработка: Аналитики проверяли статусы и категоризацию (Доход/Расход) вручную.  
Квантифицированные потери:  
- Время обработки партии: 4 часа/день.  
- Уровень человеческой ошибки при категоризации: 2.5% (250 записей требовали повторной проверки).  
  -   
- Риск комплаенс-нарушений: Отсутствие единого неизменяемого лога действий (audit trail).  
  - > не найдешь, на каком шаге произошла ошибка  


## Возможное решение  
Валидация состояния: Отсев архивных или неактивных записей согласно текущей финансовой политике (фильтрация по status).  
Трансформация данных: Приведение сырых данных к стандартизированному виду для отчетности (разделение на Income/Expense по знаку amount).  
Фиксация аудита: Запись каждого шага обработки в лог-систему для последующего аудита (требование регуляторов).  

Производительность: Обработка партии из 10 000 записей не более чем за 5 минут (снижение времени с 4 часов).  
Точность: Ошибка категоризации данных ≤ 0.1% (только технические сбои, исключена человеческая ошибка).  
Полнота аудита: 100% записей должны иметь соответствующую запись в логе обработки.  

Изменяемость правил: Бизнес-правила (критерии активности, формат отчета) должны обновляться без пересборки ядра системы (обеспечивается использованием функциональных интерфейсов).  


# Терминология и спецификация данных  

Формат данных: Структурированные записи (POJO), сериализуемые в JSON, хранящиеся в реляционной базе данных.  
Источник данных: Реляционная СУБД (PostgreSQL/MySQL) через JDBC.  
Сток данных (Sink): Лог-файл (audit log) и обновленная запись в БД.

Этап 2: Обработка ошибок и побочные эффекты (Supplier & Consumer)  

Задача: Добавить ленивую загрузку данных и логирование результатов.  
Требования:  
    Ленивая генерация данных при отсутствии кэша (использовать Supplier<T>).  
    Запись результатов в лог и отправку метрик (использовать Consumer<T> с блоком, выбрасывающим checked exception, обернутым в утилитарный интерфейс).  
Критерий отказа: Реализация через анонимные классы увеличит объем кода в 3 раза, скрыв логику бизнес-правила под слоем синтаксического шума.  

Этап 3: Конфигурация конвейера (Method & Constructor References)  

Задача: Построить DSL (Domain Specific Language) для сборки конвейера.  
Требования:  
    Использование метод-референсов для статических утилит (String::valueOf, Math::max).  
    Использование конструктор-референсов для создания коллекций результатов (ArrayList::new).  
    Композиция функций (function1.andThen(function2)).  
Критерий отказа: Без референсов придется писать лямбды-обертки (x) -> Utility.method(x), что нарушает принцип DRY (Don't Repeat Yourself) и увеличивает риск ошибки в сигнатуре типов.  

Итоговое требование к сдаче: Код должен компилироваться. В комментарии к коду указать точное количество строк, которое было бы добавлено при замене всех лямбда-конструкций на анонимные внутренние классы.
